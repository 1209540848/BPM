# BPM 项目部署与面试讲法

## 一句话定位

这是一个面向企业审批场景的 BPM 流程管理平台原型，重点实现了流程设计、流程发布、流程实例流转、任务审批、JWT 鉴权、Prisma 持久化和 WebSocket 通知能力。

不要把它包装成已经商业上线的完整企业级 BPM。更稳妥的说法是：个人主导完成的可运行 MVP，已经补齐了采购审批真实场景，并实现了条件网关与并行/自动节点推进等核心流程难点。

## 当前可演示场景

场景：采购申请审批。

- 申请人提交采购申请。
- 部门经理先审批。
- 条件网关根据 `amount` 判断金额。
- `amount < 5000`：进入财务审批后结束。
- `amount >= 5000`：追加总监审批，再进入财务审批后结束。

演示账号密码均为 `123456`：

- `applicant`：采购申请人。
- `manager`：部门经理。
- `director`：业务总监。
- `finance`：财务审批人。
- `admin`：系统管理员。

## 本地启动

### 前端

```bash
npm install
npm run dev
```

前端默认地址：`http://localhost:5173`。

### 后端

```bash
cd bpm-backend
npm install
npm run prisma:generate
npm run prisma:migrate
npm run prisma:seed
npm run dev
```

后端默认地址：`http://localhost:3000`，API 前缀为 `/api/v1`。

## 构建检查

```bash
npm run build
cd bpm-backend
npm run build
```

当前前后端构建均已通过。前端存在 chunk 体积偏大的 Vite 警告，主要来自 Ant Design Vue 和 AntV X6，后续可通过路由级分包和 manualChunks 优化。

## 推荐部署方案

### 简单演示部署

- 前端：Vercel、Netlify、静态 Nginx 均可。
- 后端：Render、Railway、云服务器均可。
- 数据库：当前项目使用 SQLite，适合演示；如果部署到无持久磁盘的平台，建议切换 PostgreSQL。

### 环境变量

后端至少需要：

```env
DATABASE_URL="file:./dev.db"
PORT=3000
NODE_ENV=production
JWT_SECRET=replace-with-a-strong-secret
JWT_EXPIRES_IN=7d
```

前端如前后端分开部署，需要配置：

```env
VITE_API_BASE_URL=https://your-backend-domain.com/api/v1
```

## 已补强的核心 BPM 难点

### 条件网关

后端新增了条件表达式判断能力，支持基于流程变量进行安全的简单表达式匹配，例如：

- `amount < 5000`
- `amount >= 5000`
- `department == "研发部"`
- `item contains "服务器"`

这解决了原来“前端能画条件网关，但后端不真正判断”的问题。

### 自动节点推进

审批任务完成后，后端会继续推进流程，自动穿透开始节点、条件网关等非人工节点，直到遇到下一个人工任务或结束节点。

### 多分支推进基础

非排他网关会把所有后续连线目标加入推进队列，具备并行分支创建多个待办的基础能力。当前更偏 MVP，尚未完整实现并行汇聚等待、复杂会签投票和补偿事务。

## 面试讲法

### 面试官问：上线了吗？

建议回答：

> 这个项目不是商业上线项目，是我个人主导实现的 BPM 原型系统。但它不是静态页面，我做了完整的前后端闭环：流程设计、流程发布、流程发起、任务审批、流程状态流转、JWT 鉴权、Prisma 持久化和 WebSocket 通知。为了让它接近真实业务，我还补了采购审批场景：小额采购走经理和财务，大额采购追加总监审批，并在后端实现了条件网关判断。

### 面试官问：这个项目最难的点是什么？

建议回答：

> 难点不是 CRUD，而是流程流转语义。我把流程定义抽象成节点和边，流程实例保存当前节点和变量。审批任务完成后，后端根据当前节点找后续边，如果遇到条件网关，会用流程变量判断表达式，比如 `amount >= 5000`，然后选择对应分支继续推进。这样可以演示同一个采购流程在不同金额下走不同审批路径。

### 面试官问：还有哪些不足？

建议回答：

> 我会主动承认它还是 MVP。当前实现了流程流转主链路，但距离成熟 BPM 还缺动态表单字段权限、组织架构、审批人规则、完整会签/或签、并行汇聚、流程版本治理、审计报表和自动化测试。我的下一步计划是先补动态表单和组织权限，因为这两个决定了 BPM 能否真正落到企业业务里。

### 面试官问：你个人主要做了什么？

建议回答：

> 我主要负责整体架构和核心闭环：前端用 Vue 3 + Pinia + Ant Design Vue + AntV X6 做流程设计和业务页面；后端用 Koa + Prisma 做接口、鉴权、流程定义、流程实例、任务审批和通知；数据库用 Prisma 建模用户、流程定义、实例、任务和历史记录。后续我重点补了条件网关和采购审批演示场景，让项目从普通后台系统更接近 BPM。

## 后续优先级

1. 动态表单：流程绑定申请表单，审批时展示业务字段。
2. 字段权限：不同审批节点配置字段可见、可编辑、必填。
3. 组织权限：用户、部门、角色、部门负责人、候选组解析。
4. 会签/或签：全部通过、任一通过、比例通过、顺序审批。
5. 并行汇聚：多个分支全部完成后再进入后续节点。
6. 测试体系：补接口测试覆盖登录、流程发布、发起、审批、条件网关。
